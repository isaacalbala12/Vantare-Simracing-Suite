## 2. Review 1 ÔÇö Code Quality (SOLID, bugs, races)

### 2.1 Race Conditions

#### ISSUE R1: `fuel/monitor.go` ÔÇö mutex declared but unused in Trigger

The `Monitor` struct declares `mu sync.Mutex` (line 69), and `SetCapacity()` uses it correctly (line 112-116). However, `Trigger()` (line 125) **never acquires the mutex** while reading/writing shared state (`m.capacity`, `m.playedHalfTank`, etc.).

```go
// SetCapacity locks:
func (m *Monitor) SetCapacity(capacity float64) {
    m.mu.Lock()
    defer m.mu.Unlock()
    m.capacity = capacity   // protected
}

// Trigger does NOT lock, but writes same fields:
func (m *Monitor) Trigger(...) []Event {
    if curr.Player.FuelCap > 0 {
        m.capacity = curr.Player.FuelCap  // UNPROTECTED write ÔÇö races with SetCapacity
    }
    ...
}
```

If `SetCapacity()` is ever called from a different goroutine than `ProcessFrame` (e.g., from a service layer on startup), this is a **data race**. The runtime's `ProcessFrame` holds `r.mu` but that doesn't protect `fuel.Monitor.mu`.

**Severity:** Low-medium ÔÇö currently no goroutine calls `SetCapacity` concurrently, but the API invites misuse.

#### ISSUE R2: `pitstops/monitor.go` ÔÇö same pattern

`mu sync.Mutex` at line 56, used only in `SetPitInfoReader()` (line 92-96). `Trigger()` (line 108) accesses `m.pitInfoReader`, `m.trackLength`, etc. without locking.

**Severity:** Low ÔÇö same as R1.

### 2.2 Wrong Telemetry Fields (Logic Bug)

#### ISSUE R3: `timings/monitor.go` ÔÇö being_held_up and being_pressured use wrong fields

The variable assignment confuses two distinct telemetry fields:

```go
gapAhead := player.TimeBehindLeader   // ÔćÉ gap to LEADER, not car ahead
gapBehind := player.TimeBehindNext     // ÔćÉ gap to NEXT CAR AHEAD, not car behind
```

- **Being held up** (line 170): checks `gapAhead < 3.0` but `gapAhead` is `TimeBehindLeader` (gap to the leader, who could be miles ahead). It should check `player.TimeBehindNext < 3.0` (gap to the car immediately ahead).
- **Being pressured** (line 181): checks `gapBehind < 2.0` but `gapBehind` is `TimeBehindNext` (gap to the car ahead). It should check the gap to the **car behind**, computed from `TimeBehindLeader` differences or by iterating the vehicle list.

**Impact:** Both events fire on incorrect conditions. "Being held up" only works when close to the leader (P2). "Being pressured" checks the wrong direction. **Both events are effectively non-functional in their current form.**

**Severity:** High ÔÇö parity claim of "being held up / pressured" is misleading; the events fire on wrong data.

### 2.3 Off-by-One / Narrow Window Issues

#### ISSUE R4: `racetime/monitor.go` ÔÇö sub-minute thresholds have fragile 6-second windows

Each time-remaining marker (20, 15, 10, 5, 2 min) uses a **6-second window**:

```go
if rem/60 < 20 && rem/60 > 19.9  // only fires when rem Ôłł (1194, 1200) ÔÇö a 6s window
```

If frame timing causes `rem` to jump from 1205 to 1190 between two ticks, the 20-minute warning is **permanently missed** (the `!m.played20Min` guard prevents re-fire). The same pattern applies to 15, 10, 5, and 2-minute markers.

**Recommended fix:** Fire when `rem` drops **below** the threshold for the first time (rising edge on threshold crossing), e.g. `rem <= 1200 && rem > prevRem` or track a `crossed20Min` boolean.

**Severity:** Medium ÔÇö likely to miss markers in real gameplay with frame drops or variable tick intervals.

#### ISSUE R5: `fuel/monitor.go` ÔÇö integer division for half-laps

```go
halfLaps := curr.Session.SessionLapsTotal / 2
if curr.Player.LapNumber >= halfLaps {
```

For odd-length races (e.g., 25 laps), `halfLaps = 12` (integer division truncates), firing at lap 12 instead of lap 12.5. Fire occurs one lap earlier than true half-distance.

**Severity:** Low ÔÇö conservative (earlier warning is safe for fuel advice).

### 2.4 Nil Safety

All monitors properly guard against nil `curr`, nil `curr.Player`, nil `curr.Session`, and nil `player` from `FindPlayerVehicle`. The runtime's `ProcessFrame` also guards against nil `frame`. **No nil-panics found** in the 12 new feature areas.

The map is checked first (line 680), so no events are actually dropped. But the switch is now stale ÔÇö a maintenance hazard.

## 7. Final Verdict

**The 22 new events are correctly implemented and wired** ÔÇö they are in the eventTextKeyMap, the runtime adapters exist, and all tests pass. Parity moves from **43% Ôćĺ 50.3%**, a gain of 7.3 percentage points. The new monitors follow the established patterns (single Trigger method, cooldowns, one-shot flags).

**However, two issues prevent the new features from being fully functional:**

1. **Timings monitor (R3 ÔÇö HIGH):** The `being_held_up` and `being_pressured` events use swapped/incompatible telemetry fields. They fire on incorrect data and are effectively non-functional. This needs a fix before these events can be relied upon.

2. **RaceTime monitor (R4 ÔÇö MEDIUM):** The 6-second window thresholds can silently drop time-remaining markers under real gameplay frame timing. A crossing-detection pattern would be more robust.

**Recommendation:** Fix R3 (being_held_up/being_pressured) and R4 (racetime thresholds) before closing the parity v2 milestone. The remaining issues (R1, R2, stale switch, R5) are low-severity and can be addressed in a maintenance pass.

**Tests:** `go test ./internal/engineer/... -count=1 -timeout 90s` ÔÇö **27/27 pass, zero failures.**
