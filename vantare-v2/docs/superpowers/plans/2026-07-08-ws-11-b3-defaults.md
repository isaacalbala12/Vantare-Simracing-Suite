# WS-11.B3 — Refactor de defaults `class*Fg` al catálogo Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move the 6 `class*Fg` defaults hardcoded in `widget-appearance.ts` (lines 32-37) into `style-catalog.ts` for ALL `vantare-crystal` entries that need them, then remove the hardcoded fallbacks. After B3, `widget-appearance.ts` falls back to the catalog (or to `GLOBAL_DEFAULTS` if neither provides a value), eliminating the duplicated source of truth.

**Critical correction from code review (P0):** The original plan assumed the catalog already had `class*Fg` fields for all widget types. It does NOT. Microcorte A3 only added them to the `relative` entry (lines 141-152 of `style-catalog.ts`). The other 5 widget types (`standings`, `delta`, `pedals`, `telemetry`, `telemetry-vertical`) are MISSING these fields. Before removing the hardcoded fallbacks from `widget-appearance.ts`, we MUST add the 6 `class*Fg` fields to all 6 `vantare-crystal` entries in the catalog. Otherwise B3 introduces a silent regression: non-`relative` widgets would fall back to `GLOBAL_DEFAULTS` (gray `#6b7280`) instead of the original hardcoded values.

**Architecture:** Two-phase refactor. Phase 1 (Tasks 2-4): extend the catalog to all 6 `vantare-crystal` entries. Phase 2 (Tasks 5-7): update `widget-appearance.ts` to fall back to `GLOBAL_DEFAULTS` only when the catalog doesn't provide a value. The hardcoded color values move entirely to the catalog.

**Tech Stack:** TypeScript, Vitest (frontend only). No Go changes.

## Design decisions

### What value to use per widget type?

The 6 `class*Fg` fields are foreground colors (text color) for class badges. They're class-specific, not widget-specific. So a `standings` widget and a `relative` widget should both use the same `classHypercarFg: "#f87171"`. The values are universal across widget types, only the catalog entry differs.

We use the same hex values that were hardcoded in `widget-appearance.ts` (the original values) for all 6 widget types:

| Field | Value (universal) |
|---|---|
| `classHypercarFg` | `#f87171` |
| `classLmp2Fg` | `#60a5fa` |
| `classLmp3Fg` | `#22d3ee` |
| `classGt3Fg` | `#fbbf24` |
| `classGt4Fg` | `#f472b6` |
| `classUnknownFg` | `#6b7280` |

### `GLOBAL_DEFAULTS` after the refactor

`GLOBAL_DEFAULTS` becomes a true safety net for unknown widget types or missing entries. We use the same universal values. These are reached only if:
- A widget type not registered in `WIDGET_TYPES` is passed.
- The catalog entry is missing the field (e.g. `vantare-racing` entries don't have `class*Fg` because they predate A3).

`GLOBAL_DEFAULTS` is identical to the hardcoded values being removed. This is intentional: the system is in a better state because the catalog is the primary source, and the global defaults match the catalog values. There's no behavior change for any widget that has a catalog entry.
