### Escenario 1: Usuario pagó pero la app muestra Free

**Síntomas:** El usuario reporta que hizo el pago en Stripe pero la app sigue en Free, con banner "Sin suscripción".

**Diagnóstico:**

```bash
./vantare-admin lookup <email>
```

Verificar que `user_entitlements` tiene fila con `status: active` para el product_key correcto.

**Causas posibles:**

1. **Webhook no procesado** — el pago en Stripe no llegó a Supabase.
   - Ir a Stripe Dashboard → Webhooks → ver eventos recientes para `checkout.session.completed`.
   - Si el evento existe y tiene `200 OK` pero no hay entitlement, revisar logs de la EF: `supabase functions logs stripe-webhook`.
   - Si el evento no existe o tiene error, reenviar desde Stripe Dashboard (botón "Resend").

2. **El RPC get_account_entitlements no devuelve el entitlement** — el Go valida pero no lo ve.
   - Ejecutar desde Supabase SQL Editor:
     ```sql
     select * from get_account_entitlements('<fingerprint>');
     ```
   - Si no devuelve el entitlement, revisar que `user_entitlements.user_id` coincida con `profiles.id` y que `status = 'active'`.

3. **Device-limit** — el usuario registró otro PC antes y este está bloqueado.
   - El lookup muestra `active_device` distinto al fingerprint actual.
   - Resolución: `./vantare-admin device-reset <email>` (fuerza reset ignorando rate-limit).

**Resolución de emergencia (otorgar entitlement manualmente):**

```bash
./vantare-admin grant <email> <product_key>
```

Esto inserta o reactiva el entitlement en `user_entitlements`. El usuario debe reabrir la app para revalidar.

### Escenario 2: Usuario atrapado en device-limit

**Síntomas:** La app muestra "Límite de dispositivos" y no deja entrar.

**Diagnóstico:**

```bash
./vantare-admin lookup <email>
```

Verificar `active_device` != fingerprint del usuario.

**Resolución:**

```bash
./vantare-admin device-reset <email>
```

Esto limpia `devices.fingerprint_hash` y `devices.last_reset_at`. El usuario debe cerrar y reabrir la app; en el próximo `Validate`, el Go registrará su fingerprint como el activo.

Si el rate-limit (1 reset/24h) bloquea el RPC `reset_active_device`, usar el CLI que ignora el rate-limit (va directo a la tabla con service-role).

---

### Escenario 3: Reembolso o cancelación

**Síntomas:** El usuario pide reembolso o canceló y la app sigue mostrando plan activo.

**Reembolso (Stripe):**
1. Ir a Stripe Dashboard → Payments → buscar el pago.
2. Clic "Refund" → confirmar importe.
3. Stripe emite `charge.refunded` (no manejado por la EF → el entitlement sigue activo).
4. **Manual:** revocar entitlement tras reembolso:
   ```bash
   ./vantare-admin revoke <email> <product_key>
   ```

**Cancelación de suscripción (Stripe Dashboard):**
1. Ir a Customers → buscar usuario → Subscriptions.
2. Cancelar (inmediata o al final del período).
3. Stripe emite `customer.subscription.deleted` → la EF lo procesa y marca `status = 'expired'`.
4. Verificar en Supabase: `select * from user_entitlements where user_id = '<id>'`.
5. Si no se procesó, revocar manualmente con el CLI.

### Logs y auditoría

Para auditar el historial de un usuario:

```bash
./vantare-admin events <email>
```

Esto lista los últimos 20 `license_events` (validación, webhook, device reset, grace enter/expire).

Para logs en vivo de la Edge Function:

```bash
supabase functions logs stripe-webhook --project-ref <ref>
```
