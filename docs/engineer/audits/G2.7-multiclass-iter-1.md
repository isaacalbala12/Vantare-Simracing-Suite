## Material differences persisting

1. **Zonas de distancia variables por track length (medio impacto)**: CC ajusta zonas según track length (5 clases: VERY_SHORT..VERY_LONG). Vantare usa zonas fijas. **Implementar en iter-2** con TrackLengthClass lookup.

2. **canAnnounce() con cooldowns por tipo (medio impacto)**: CC tiene cooldowns separados para single vs multiple, faster vs slower (45-60s). Vantare usa 4s+6s, que es mucho más permisivo. **Alinear en iter-2**.

2. **Telemetry requerido (PARITY_BLOCKED parcial)**: CC usa `OpponentData.DistanceRoundTrack`, `OpponentData.Speed`, `OpponentData.CurrentBestLapTime`, `OpponentData.CarClass`, `OpponentData.ClassPosition`, `OpponentData.OverallPosition`, `OpponentData.CompletedLaps`. Vantare tiene `vehicleScoring.LapDistance` (≈distanceRoundTrack) y `VehicleClass` (string), pero no tiene `CarClassEnum` mapeado ni `ClassPosition`, `OverallPosition`, `BestLapTime`, `Speed` por vehículo en el modelo de telemetría engineer.
   - **`DistanceRoundTrack`** → Vantare tiene `LapDistance` (float64) en `VehicleScoring`. **MATCH**.
   - **`CarClass`** → Vantare tiene `VehicleClass` (string). Sin embargo CC usa un enum `CarClassEnum` mapeado a sonidos. Vantare necesitaría mapear strings LMU → clasificación. **PARITY_BLOCKED** hasta que esté el mapeo.
   - **`ClassPosition`** → No disponible en `VehicleScoring`. LMU Shared Memory lo expone como `mClassPosition`. **PARITY_BLOCKED**.
   - **`OverallPosition`** → `Place` en Vantare, pero solo para el player. Para oponentes, `Place` es su overall position. **MATCH**.
   - **`BestLapTime`** → No disponible en Vantare `VehicleScoring`. LMU SM expone `mBestLapTime`. **PARITY_BLOCKED**.
   - **`Speed`** → No disponible por vehículo en Vantare. LMU SM expone `mSpeed`. **PARITY_BLOCKED**.

3. **Zonas de distancia variables (medio impacto)**: CC ajusta las zonas de warning según track length (short/normal/long). Vantare no tiene `TrackDefinition.trackLengthClass`. **Implementar con track length lookup** cuando esté disponible.

4. **canAnnounce() con cooldowns por tipo (bajo impacto)**: CC tiene cooldowns separados para single vs multiple, faster vs slower (45-60s). **Implementar en iter-2**.

## Non-material differences (accepted)

- Vantare no tiene sistema de voice commands para "what class is car ahead/behind". Se implementará en fase de voice.
- Vantare no tiene 23 car class sound mappings (depende del voice pack). Se usará texto genérico inicialmente.

## Plan para iter-2 (mejoras)

1. Añadir detección de "fighting" usando `maxSeparateToBeConsideredFighting = 30m` entre oponentes del mismo lap.
2. Añadir detección de class leader (`ClassPosition == 1`).
3. Añadir detección de racing-for-position (`OverallPosition` diff == 1).
4. Implementar zonas variables según `TrackLengthClass`.
5. Implementar minLaps/minTime gates por track length class.
6. Añadir cooldown más largo entre anuncios del mismo tipo (45-60s).
7. Añadir `forceSingleClass` gate.
8. Marcar `PARITY_BLOCKED` para car class enum mapping completo (23 clases).

## Bloqueos

- `telemetry.VehicleScoring` no tiene: `VehicleClass` (string exists pero sin enum LMU mapping), `ClassPosition`, `Speed`, `BestLapTime`.
- LMU Shared Memory expone estos campos pero no están mapeados en el parser de Vantare.
- Sin `OpponentData` map keyed by class, no podemos determinar "faster" vs "slower" classes sin `BestLapTime` o `CarClassEnum`.
- `TrackDefinition.trackLengthClass` no existe en Vantare.
